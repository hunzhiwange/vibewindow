# Static Canvas Support Boundaries

VibeWindow's Tailwind canvas is a static design surface, not a browser runtime. The current behavior is defined by four code paths:

- `TailwindParser::parse` turns class strings into a static `ParsedStyle` snapshot.
- `TailwindParser::analyze` classifies each original token as `FlattenedVariant`, `ExportOnly`, or `Unsupported` for canvas-facing tooling.
- `renderer.rs` and `renderer/layout.rs` consume that snapshot for canvas layout, paint, hit-test, and bounds.
- `import.rs` keeps the original class string on the imported design element and copies a subset of parsed values into design fields.
- `export.rs` keeps emitting the original class string while also generating inline fallback styles from the design element.

This means every Tailwind capability currently falls into one of the following buckets.

## 1. Rendered On Canvas

These utilities can participate in the static canvas preview and geometry today when they are represented in `ParsedStyle` and consumed by the renderer:

- Box model and sizing: `w-*`, `h-*`, `w-full`, `h-full`, padding, margin, gap.
- Basic paint: background color, border width, side borders, border color, border radius, border style, divide utilities.
- Basic effects: `outline-*` width/style/color/offset and `shadow-sm|shadow|shadow-md|shadow-lg` as static canvas effects.
- Static layout switches: `block`, `inline`, `inline-block`, `flex`, `inline-flex`, `grid`, `hidden`.
- Partial container layout: flex row/column, basic align/justify, basic grid columns, partial absolute positioning.
- Static assets: `img`, `svg`, and simple text blocks.
- Text preview subset: text color, font size, text alignment, and logical start/end alignment when paired with `dir`.

Support here means "the static canvas can draw a snapshot". It does not mean the browser runtime semantics are complete.

## 2. Export-Preserved Only

The original Tailwind class string is preserved on `element.class` during import and written back during export, even when the canvas cannot use it.

Typical examples:

- Utilities that are recognized by import/export tooling but are not fully consumed by the canvas renderer.
- Classes that should stay visible in exported HTML for downstream Tailwind processing.
- Tokens that are useful as authoring metadata even if the design canvas cannot faithfully preview them.

For this bucket, the contract is: the class text survives round-trip export, but it must not be assumed to affect canvas layout or paint.

## 3. Explicitly Degraded Or Unsupported

The following capabilities do not have real runtime behavior on the static canvas and must be treated as degraded or unsupported:

- Interactive states such as `hover:`, `focus:`, `active:`, `visited:`, `disabled:`, `group-*`, `peer-*`.
- Runtime responsive behavior such as `sm:`, `md:`, `lg:`, `xl:`, `2xl:`.
- Theme or environment variants such as `dark:`.
- Animation and transition families such as `animate-*`, `transition-*`, `duration-*`, `ease-*`.
- Complex visual effects such as filter, backdrop-filter, ring, and browser-dependent shadow variants that are not mapped into the canvas renderer.

Current effect notes:

- Outline support is static only: it expands canvas bounds and hit-test, but does not change layout flow.
- Shadow support is limited to the named preset subset above and renders as a simplified static silhouette, not a browser-accurate blur stack.
- `shadow-none` clears the simplified shadow; unsupported shadow tokens remain export-preserved only.

Current downgrade behavior:

- Responsive and interaction variants that can be safely flattened are recorded as `FlattenedVariant` and applied as a static snapshot only.
- `dark:` is currently `ExportOnly`: the canvas preview does not switch Tailwind semantics with theme mode.
- Known advanced-effect families such as `animate-*`, `filter-*`, `backdrop-*`, `mask-*`, complex `shadow-*`, and `ring-*` are `ExportOnly`.
- Unknown or unmatched tokens are recorded as `Unsupported` instead of being treated as an implicit success path.
- In every case above, the original class string is still preserved for export.

Examples:

- `md:flex` can appear as `flex` in the static preview, but there is no breakpoint-aware layout on canvas.
- `hover:bg-blue-500` can appear as a plain background color preview, but there is no hover state.
- `dark:bg-slate-900` stays in the exported class list, but does not override the current static preview snapshot.
- `animate-spin` is preserved in the exported class list, but the canvas does not animate.

## Downgrade Policy

When a capability cannot be represented safely as a static canvas rule, the preferred behavior is:

1. Keep the original class text for import/export round-trip.
2. Do not silently invent partial runtime semantics in the UI layer.
3. Reuse `TailwindParser::analyze` for future unsupported-token UI instead of re-implementing variant/effect checks in view code.

This document is the source of truth for the current boundary between static preview, export-only preservation, and unsupported Tailwind behavior.
