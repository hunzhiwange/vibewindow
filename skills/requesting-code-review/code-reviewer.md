## 审查清单

**代码质量：**
- 关注点是否清晰分离？
- 错误处理是否得当？
- 类型安全性（如适用）？
- 是否遵循 DRY（不重复）原则？
- 边界情况是否已处理？

**架构：**
- 设计决策是否合理？
- 是否考虑了可扩展性？
- 性能影响如何？
- 是否存在安全隐患？

**测试：**
- 测试是否真正测试了逻辑（而非仅模拟）？
- 边界情况是否已覆盖？
- 是否在需要的地方进行了集成测试？
- 所有测试是否通过？

**需求：**
- 计划中的所有需求是否已满足？
- 实现是否与规范一致？
- 是否存在范围蔓延？
- 破坏性变更是否已记录？

**生产就绪程度：**
- 迁移策略（如有 schema 变更）？
- 是否考虑了向后兼容性？
- 文档是否完整？
- 是否存在明显的 bug？

## 关键规则

**应该做的：**
- 按实际严重程度分类（不是所有问题都是"严重"）
- 提供具体信息（文件:行号，而非模糊描述）
- 解释问题为何重要
- 肯定做得好的地方
- 给出明确的结论

**不应该做的：**
- 不检查就说"看起来不错"
- 将吹毛求疵的问题标记为"严重"
- 对未审查的代码给出反馈
- 使用模糊表述（如"改进错误处理"）
- 回避给出明确结论
