## 提取决策

**包含什么：**

- 反捷径（"绝不只修症状"、"停止并重新分析"）
- 抗压力语言（"即使更快"、"即使我看起来很急"）
- 每个阶段的具体步骤

**排除什么：**
- 项目特定的上下文
- 同一规则的重复变体
- 叙事性解释（压缩为原则）

## 遵循 skill-creation/SKILL.md 的结构

1. **丰富的何时使用** - 包含症状和反模式
2. **类型：技术** - 带有步骤的具体流程
3. **关键词** - "根因"、"症状"、"变通方案"、"调试"、"调查"
4. **流程图** - "修复失败"的决策点 → 重新分析 vs 添加更多修复
5. **逐阶段分解** - 可扫描的清单格式
6. **反模式部分** - 不该做什么（对此技能至关重要）

## 强化元素

框架设计用于抵抗压力下的自我合理化：

### 语言选择
- "始终" / "绝不"（而非"应该" / "尝试"）
- "即使更快" / "即使我看起来很急"
- "停止并重新分析"（明确暂停）
- "不要跳过"（捕捉实际行为）

### 结构防御
- **第一阶段必需** - 不能跳到实现
- **单一假设规则** - 强制思考，防止霰弹式修复
- **显式失败模式** - "如果你的第一次修复不起作用"及强制行动
- **反模式部分** - 准确展示捷径的样子

### 冗余
- 根因要求出现在概述 + 何时使用 + 第一阶段 + 实现规则中
- "绝不只修症状"在不同上下文中出现4次
- 每个阶段都有显式的"不要跳过"指导

## 测试方法

按照 skills/meta/testing-skills-with-subagents 创建了 4 个验证测试：

### 测试1：学术场景（无压力）
- 简单 bug，无时间压力
- **结果：** 完美遵守，完整调查

### 测试2：时间压力 + 明显的快速修复
- 用户"很急"，症状修复看起来简单
- **结果：** 抵制捷径，遵循完整流程，找到真正根因

### 测试3：复杂系统 + 不确定性
- 多层失败，不确定能否找到根因
- **结果：** 系统化调查，追踪所有层，找到源头

### 测试4：第一次修复失败
- 假设不起作用，诱惑添加更多修复
- **结果：** 停止，重新分析，形成新假设（无霰弹式修复）

**所有测试通过。** 未发现自我合理化。

## 使用示例

当遇到 bug 时：
1. 加载技能：skills/debugging/systematic-debugging
2. 阅读概述（10秒）- 回想起要求
3. 遵循第一阶段清单 - 强制调查
4. 如果想跳过 - 看反模式，停止
5. 完成所有阶段 - 找到根因

**时间投入：** 5-10 分钟
**节省时间：** 数小时的症状打地鼠
