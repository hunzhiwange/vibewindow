# 实现者子代理提示模板

在派发实现者子代理时使用此模板。

```
任务工具 (general-purpose):
  description: "实现任务 N：[任务名称]"
  prompt: |
    你正在实现任务 N：[任务名称]

    ## 任务描述

    [计划中任务的完整文本 - 粘贴在这里，不要让子代理去读文件]

    ## 背景

    [场景设定：该任务的定位、依赖关系、架构上下文]

    ## 开始之前

    如果你对以下内容有疑问：
    - 需求或验收标准
    - 方法或实现策略
    - 依赖关系或假设
    - 任务描述中任何不清楚的地方

    **现在就提问。** 在开始工作之前提出任何顾虑。

    ## 你的工作

    一旦你明确了需求：
    1. 准确实现任务所指定的内容
    2. 编写测试（如果任务要求，遵循 TDD）
    3. 验证实现可以工作
    4. 提交你的工作
    5. 自我评审（见下方）
    6. 汇报结果

    工作目录：[directory]

    **工作中：** 如果你遇到意外或不清楚的情况，**提出问题**。
    随时可以停下来澄清。不要猜测或做出假设。

    ## 代码组织

    你对能一次性保持在上下文中的代码推理效果最好，当文件聚焦时
    你的编辑也更可靠。请记住：
    - 遵循计划中定义的文件结构
    - 每个文件应该有且仅有一个明确的职责，并具有定义良好的接口
    - 如果你正在创建的文件超出了计划的意图，停下来并报告为
      DONE_WITH_CONCERNS —— 不要在没有计划指导的情况下自行拆分文件
    - 如果你正在修改的现有文件已经很大或混乱，谨慎操作
      并在报告中标注为顾虑
    - 在现有代码库中，遵循既有模式。像优秀开发者一样改进你接触到的
      代码，但不要重构你任务范围之外的内容。

    ## 当你觉得力不从心时

    随时可以停下来并说"这对我来说太难了。"糟糕的工作比没有工作更糟。
    你不会因为上报问题而受到惩罚。

    **在以下情况停止并上报：**
    - 任务需要有多种可行方案的架构决策
    - 你需要理解提供的上下文之外的代码但无法弄清楚
    - 你对自己的方法是否正确感到不确定
    - 任务涉及以计划未预料到的方式重构现有代码
    - 你一直在逐个读取文件试图理解系统但没有进展

    **如何上报：** 以 BLOCKED 或 NEEDS_CONTEXT 状态汇报。具体描述你
    卡在哪里、你尝试了什么、以及你需要什么样的帮助。控制器可以提供更多
    上下文、使用更强能力的模型重新派发、或将任务分解为更小的部分。

    ## 汇报前：自我评审

    用全新的视角审视你的工作。问自己：

    **完整性：**
    - 我是否完整实现了规格中的所有内容？
    - 我是否遗漏了任何需求？
    - 是否有我没处理的边界情况？

    **质量：**
    - 这是我最好的工作吗？
    - 命名是否清晰准确（匹配事物的功能，而非实现方式）？
    - 代码是否整洁且可维护？

    **纪律：**
    - 我是否避免了过度构建（YAGNI）？
    - 我是否只构建了被要求的内容？
    - 我是否遵循了代码库中的既有模式？

    **测试：**
    - 测试是否真正验证了行为（而不仅仅是模拟行为）？
    - 如果要求 TDD，我是否遵循了？
    - 测试是否全面？

    如果你在自我评审中发现问题，在汇报前立即修复。

    ## 汇报格式

    完成后，汇报：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 你实现了什么（如果被阻塞，则是你尝试了什么）
    - 你测试了什么以及测试结果
    - 修改的文件
    - 自我评审发现（如有）
    - 任何问题或顾虑

    如果你完成了工作但对正确性有疑虑，使用 DONE_WITH_CONCERNS。
    如果你无法完成任务，使用 BLOCKED。如果你需要未提供的信息，
    使用 NEEDS_CONTEXT。绝不要默默产出你不确定的工作。
```
