# AGENTS.md

本文件定义本仓库编码代理的默认工作协议，范围为整个仓库。

## 项目概览

VibeWindow 是以 Rust 为先的自主代理运行时，优化目标为高性能、高效率、高稳定性、高可扩展性、高可持续性、高安全性。

## 工程原则

这些原则默认强制执行。它们不是口号，而是实现约束。

- 优先使用直白控制流，避免炫技的元编程。
- 优先使用显式 match 分支与强类型结构体，避免隐藏的动态行为。
- 保持错误路径清晰且局部化。
- 没有明确已接受的用例，不新增配置键、trait 方法、特性开关或流程分支。
- 没有至少一个当前调用者，不引入“为未来准备”的抽象。
- 对不支持的路径保持显式（直接报错），而不是添加部分伪支持。
- 当有助于清晰性时，允许复制小而局部的逻辑。
- 只有在出现稳定的重复模式后才提取共享工具（三次原则）。
- 提取时保持模块边界，避免隐藏耦合。
- 每个模块聚焦单一关注点。
- 尽可能通过实现现有的窄 trait 来扩展行为。
- 避免把策略 + 传输 + 存储混在一起的胖接口与“上帝模块”。
- 对不支持或不安全状态优先显式 `bail!`/错误。
- 绝不静默扩大权限/能力。
- 当回退是有意且安全时，记录回退行为。
- 访问与暴露边界采用默认拒绝。
- 绝不记录密钥、原始令牌或敏感载荷。
- 将网络/文件系统/命令行范围尽量收窄，除非有明确正当理由。
- 保持变更易于回滚（小范围、清晰爆炸半径），避免混合型大补丁阻碍安全回滚。
- 将文档视为一等产品面，不是合并后的附属物。
- 将注释视为一等产品面，不是合并后的附属物。


## 代理工作流

1. 先读后写
    - 编辑前检查现有模块、工厂连线与相邻测试。
2. 定义范围边界
    - 一个 PR 一个关注点；避免混合特性 + 重构 + 基础设施补丁。
3. 实现最小补丁
4. 按风险等级验证
    - 仅文档：轻量检查。
    - 代码/高风险变更：完整相关检查与聚焦场景。

### 代码命名契约

除非某子系统已有更强模式，否则所有代码变更必须遵循以下命名规则：

- 统一使用 Rust 标准命名：模块/文件 `snake_case`，类型/trait/枚举 `PascalCase`，函数/变量 `snake_case`，常量/静态 `SCREAMING_SNAKE_CASE`。
- 类型与模块按领域职责命名，而非实现细节（例如 `SecurityPolicy` 优于 `Manager`）。
- Trait 实现者命名清晰可预测：`<ProviderName>Provider`。
- 工厂注册 key 保持稳定、小写、面向用户（例如 `"shell"`），无迁移需要时避免别名扩散。
- 测试按行为/结果命名（`<subject>_<expected_behavior>`），fixture 标识符保持中性、项目范围。
- 每个 Rust 文件尽可能只做一件事，文件大小最好不要超过500行，超过时考虑拆分。

### 架构边界契约

用以下规则保持 trait/工厂架构在增长中稳定：

- 扩展能力优先通过新增 trait 实现 + 工厂连线；避免为孤立功能进行跨模块重写。
- 依赖方向指向契约内聚：具体集成依赖 trait/config/util 层，而不是其他具体集成。
- 避免创建跨子系统耦合（例如 provider 代码导入 channel 内部，tool 代码直接修改 gateway 策略）。
- 模块职责单一。
- 仅在重复出现后引入共享抽象（三次原则），且当前范围至少有一个真实调用者。

## 验证矩阵

代码变更的默认本地检查：

```bash
任务执行后，不要长篇大论地总结，只需要总结出任务执行的结果即可，50字以内。
一定不要执行这个命令 cargo fmt --all -- --check，除非用户明确要求执行
一定不要执行这个命令 cargo check --target wasm32-unknown-unknown 除非用户明确要求执行
一定不要执行这个命令 cargo test 除非用户明确要求执行
一定不要执行 Makefile 相关命令，除非用户明确要求执行
代码写完成后，一定要执行基础检测 cargo clippy --all-targets --all-features -- -D warnings，如果有错误和警告，需要修复问题，如果发现别的AI改变影响此次编译，先不要管。
单元测试文件必须单独放到一个独立文件，不能和逻辑放到一起。
对于 Iced GUI framework UI 相关功能，特别注意要适配暗黑主题，避免在暗黑主题下出现文字、图标、背景颜色不协调导致看不清，注意容器垂直居中和水平居中，只要是滚动条宽度都是4px。
```

## 反模式

- 不要为小便利添加重型依赖。
- 不要悄然削弱安全策略或访问约束。
- 不要添加“以防万一”的臆测配置/特性开关。
- 不要将大规模纯格式改动与功能改动混在一起。
- 不要“顺手”修改无关模块。
- 不要在没有明确说明的情况下绕过失败检查。
- 不要在重构提交中隐藏行为变化的副作用。
- 除非维护者在当前范围明确要求，不要尝试仓库重品牌/身份替换。

## 编码护栏

在快速迭代模式中：

- 保持每次迭代可逆（小提交、清晰回滚）。
- 实现前用代码搜索验证假设。
- 优先确定性行为，避免机巧捷径。
- 不要在安全敏感路径上“先上再说”。
- 不确定时，留下带验证上下文的明确 TODO，而不是隐藏猜测。
