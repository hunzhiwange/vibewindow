# 代码健康分析

## 概述

对代码库执行安全和架构双维度的健康检查，生成结构化的 Markdown 报告。分析完成后将报告保存至 `docs/health-reports/YYYY-MM-DD-code-health.md`。

## 分析流程

### 阶段一：安全检查

按以下顺序执行，任一命令失败不阻塞后续步骤：

1. **依赖漏洞扫描**
   ```
   cargo deny check advisories
   ```
   记录：漏洞数量、严重等级、受影响 crate、是否有修复版本

2. **依赖许可合规**
   ```
   cargo deny check licenses
   ```
   记录：未经允许的许可证、新增的受限依赖

3. **依赖来源审计**
   ```
   cargo deny check sources
   ```
   记录：未知注册源、Git 依赖、非 crates.io 来源

4. **代码安全模式扫描**
   - 搜索 `unsafe` 块：记录文件位置和用途
   - 搜索 `unwrap()` / `expect()`：标记可能 panic 的调用点
   - 搜索 `.env` / `token` / `secret` / `password` / `api_key`：检查敏感信息泄露风险
   - 搜索 `#[allow(`：标记被抑制的 lint 警告

### 阶段二：架构分析

1. **Clippy 静态分析**
   ```
   cargo clippy --all-targets --all-features -- -D warnings 2>&1
   ```
   统计警告/错误数量，分类汇总

2. **模块结构分析**
   - 列出 `crates/` 下所有 crate 及职责
   - 检查 `Cargo.toml` 中依赖方向：是否存在循环依赖
   - 检查 `AGENTS.md` 中约定的架构原则遵守情况：
     - 模块是否专注单一关注点
     - 是否存在"上帝模块"（过大的文件/模块）
     - trait 实现者命名是否可预测

3. **代码规模与复杂度**
   - 统计各 crate 的源码行数
   - 标记超过 500 行的单文件
   - 标记超过 20 个方法的 impl 块

4. **测试覆盖**
   ```
   cargo test --workspace --lib --bins 2>&1
   ```
   记录：测试总数、通过/失败数、编译耗时

### 阶段三：生成报告

按以下模板生成 Markdown 报告：

```markdown
# 代码健康报告 — YYYY-MM-DD
## 概览
| 维度 | 状态 | 说明 |
|------|------|------|
| 安全 | 🟢/🟡/🔴 | 简要说明 |
| 架构 | 🟢/🟡/🔴 | 简要说明 |
| 编译 | 🟢/🟡/🔴 | 简要说明 |
| 测试 | 🟢/🟡/🔴 | 简要说明 |
## 安全
### 依赖漏洞
### 许可与来源
### 代码安全模式
## 架构
### 模块结构
### 架构原则合规
### 代码规模
### Clippy 警告
## 测试
## 建议
### 紧急
### 短期
### 长期
```

## 状态判定规则

| 状态 | 条件 |
|------|------|
| 🟢 健康 | 无漏洞、0 clippy 警告、测试全通过 |
| 🟡 注意 | 有 warning 级别问题、有可延后处理的建议 |
| 🔴 危险 | 有 error 级别问题、有已知漏洞、测试失败 |

## 注意事项

- 所有命令在项目根目录执行
- 如果 `cargo deny` 未安装，提示用户执行 `cargo install cargo-deny`
- 报告保存后向用户展示关键发现的摘要
- 不修改任何代码，仅输出分析结果
