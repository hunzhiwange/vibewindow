# Lunaris 风格提示

## 一句话定义

Lunaris = 深色、克制高亮、偏未来技术界面的 AI 产品与开发者工具风格。

## 视觉气质

- 以深石墨背景和高对比浅文字建立秩序感，整体冷静、清晰、偏系统界面。
- 主强调色优先使用信号感橙色，辅以冷色信息态与少量紫色标签，不做大面积霓虹铺满。
- 层级依靠描边、面板分层、信息密度和节奏表达，不依赖夸张阴影。
- 发光和高亮必须节制，只打在按钮、焦点、状态或关键数据节点上。
- 画面更像 AI 工作台、模型控制台、任务运行面板，而不是温暖亲和营销页。

## 版式规则

- 优先使用清晰的产品壳层结构：首屏预览、工作台布局、面板分区、列表与日志区域。
- 首屏适合“标题区 + 状态标签 + 双 CTA + 产品预览面板”。
- 中段适合能力矩阵、集成流程、日志片段、数据表、环境配置区。
- 模块间留白要稳定，模块内分区要明确，避免暗背景下大量浮层互相抢焦点。
- 桌面端优先用宽容器与双栏或三段式信息结构，移动端再收敛成单栏。

## 组件语义

- Button：主按钮用于核心命令动作，次按钮用于补充动作，Outline/Ghost 用于工具条和行内动作。
- Card：用于面板、预览窗、运行摘要、能力区块，默认按 Header / Content / Actions 理解。
- Search Box / Input Group / Select Group：用于查询、配置、筛选、模型与环境选择。
- Tabs：只做模块内视图切换，不替代全局导航。
- Sidebar：只在工作台、控制台、文档中心这类产品壳层使用。
- Alert：用于短状态反馈、告警、风险提示，不用于大段说明。
- Data Table：用于任务、日志、评测、版本、权限、集成等结构化数据。
- Label / Icon Label：用于状态、版本、能力、环境标签，不承担操作按钮职责。

## 排版规则

- 标题可以直接使用技术词汇、能力名、模型名、系统状态名，但必须短而清楚。
- 正文和说明优先克制，避免过度营销化和情绪化语气。
- 主体文案优先使用 `$--font-secondary`，保证深色界面的长期阅读舒适度。
- `$--font-primary` 适合代码样式标签、版本号、模型 ID、命令片段、数值信号，不要整页大段滥用。
- 弱说明、辅助信息、日志次要字段优先使用 `$--muted-foreground`。

## 常用 token

- 背景：`$--background`
- 主文字：`$--foreground`
- 面板：`$--card`
- 面板文字：`$--card-foreground`
- 主操作：`$--primary`
- 次级底色：`$--secondary`
- 弱文字：`$--muted-foreground`
- 描边 / 输入边界：`$--border` / `$--input`
- 焦点：`$--ring`
- 侧边栏：`$--sidebar` / `$--sidebar-accent`
- 成功 / 警告 / 错误 / 信息：`$--color-success` / `$--color-warning` / `$--color-error` / `$--color-info`
- 圆角：`$--radius-m` / `$--radius-pill`
- 字体：`$--font-primary` / `$--font-secondary`

## 风格继承 JSON 说明

- `lunaris/theme` 只保留继承 Lunaris 所需的深色核心 token：背景、面板、文字、主操作、弱化层级、侧边栏、状态色、字体、圆角。
- 该 JSON 不是完整 demo，而是给 AI 的最小风格继承面，目的是在低 token 情况下仍保留 Lunaris 的深色技术气质。
- `lunaris/components` 只保留最关键的基础组件骨架、状态变体、继承关系、使用场景和覆盖规则。
- 生成时优先依赖 `lunaris/THEME.md`、`lunaris/DESIGN.md`、`lunaris/theme.json`、`lunaris/components.json`，不要把完整 `lunaris.json` 当首选提示材料。

## 适合的模块

- AI 产品首屏
- 模型能力矩阵
- 工作流步骤与集成链路
- 控制台预览
- 运行日志与任务状态
- API / SDK / 权限配置
- 帮助中心入口与系统设置页

## 不适合的模块

- 温暖亲和的消费品首页
- 儿童、社区、生活方式类软性内容页
- 依赖大面积暖色插画的品牌叙事页
- 以情绪转化为主的轻营销收口页
- 低对比、浅层级、弱边界的极简海报式布局
